# Bot-to-Bot 网站测试协作工作流

## 场景

两个飞书bot协作测试一个网站：
- **服务器端bot**：部署在网站所在服务器，负责确认服务状态、发送凭据
- **客户端bot**：从外部访问网站，执行功能测试

## 工作流程

### 第一阶段：服务器端确认

服务器端bot检查：
```bash
# 服务状态
systemctl is-active stock-web
systemctl is-active nginx

# 端口监听
ss -tlnp | grep -E '80|443|8080'

# API可用性
curl -s -o /dev/null -w "%{http_code}" http://localhost/api/concepts

# 页面可达性
curl -s -o /dev/null -w "%{http_code}" http://localhost/concept
```

确认正常后，通过飞书发送登录凭据给客户端bot。

### 第二阶段：客户端获取凭据

**⚠️ 关键陷阱**：轮询守护进程的日志会截断长消息（~260字符/行）。

**正确方式**：直接调用飞书API获取完整消息内容（见SKILL.md中的代码示例）。

**错误方式**：依赖 `grep` 轮询日志文件 → 凭据会被截断。

### 第三阶段：客户端测试

1. **登录**：用浏览器工具登录网站
   - 如有验证码，用 vision 工具识别
   - CAPTCHA识别流程：截图 → vision分析 → 输入验证码 → 点击登录
   - 如果识别错误，页面会刷新验证码，重复上述流程

2. **逐页测试**：点击导航栏的每个链接
3. **记录结果**：每个页面记录：
   - URL路径
   - HTTP状态码
   - 页面内容是否正常
   - 如有错误，截图保存

4. **API vs 页面渲染区分**：
   ```bash
   # 检查API端点
   curl -s -o /dev/null -w "%{http_code}" "https://site.com/api/concepts"
   # 检查页面端点
   curl -s -o /dev/null -w "%{http_code}" "https://site.com/concept"
   ```

   | API状态 | 页面状态 | 问题定位 |
   |---------|---------|---------|
   | 200 | 200 | ✅ 正常 |
   | 200 | 500 | 前端渲染/模板问题 |
   | 500 | 500 | 后端数据/逻辑问题 |
   | 200 | 302 | 未登录重定向（检查cookie） |
   | 404 | 404 | 路由未配置 |

5. **即时通知**：每发现一个问题就立即通知服务器端，不要等全部测完。

### 第四阶段：服务器端修复

服务器端收到报告后：
1. 检查应用日志：`journalctl -u stock-web -n 50`
2. 检查nginx日志：`tail -50 /var/log/nginx/error.log`
3. 修复后通知客户端重新测试

## 常见问题

### 验证码干扰
- 网站有验证码时，需要用 vision 工具识别验证码文字
- 验码识别失败时，刷新页面获取新验证码
- **CAPTCHA识别流程**：截图 → vision分析 → 输入验证码 → 点击登录
- 如果识别错误，页面会刷新验证码，重复上述流程

### Session过长
- 浏览器session可能超时
- 超时后需要重新登录

### 消息截断
- 轮询日志截断长消息
- 解决：用飞书API直接获取完整消息

## 第五阶段：修复验证闭环

服务器端修复后，客户端必须重新验证：

1. **curl预检**：先用curl确认页面不再返回500
   ```bash
   curl -s -o /dev/null -w "%{http_code}" "https://site.com/concept"
   # 302=未登录重定向(正常), 200=页面OK, 500=仍未修复
   ```

2. **浏览器验证**：重新登录 → 访问修复的页面 → 截图确认
3. **功能验证**：不仅检查页面能打开，还要验证核心功能（搜索、筛选、数据展示）
4. **发送验证报告**：明确列出每项测试结果（✅/❌）

**修复验证报告模板**：
```
概念板块验证完成 ✅：
1) 页面加载正常，显示493个概念
2) 搜索功能正常（测试DeepSeek→1个结果）
3) 概念卡片显示名称、代码、股票数量
4) 视觉效果良好。修复成功！
```

## 实战案例：display_date缺失导致500

**问题**：`/concept`页面返回500，但`/api/concepts`返回200
**根因**：模板第1127行引用了`display_date`字段，但数据中没有该字段
**修复**：服务器端修改模板，移除或兼容该字段
**验证**：curl预检(302) → 浏览器登录 → 访问/concept → 搜索功能测试 → 截图确认

**关键教训**：
- API正常但页面500 → 一定是前端模板/渲染问题，不是数据问题
- 通知服务器端时明确指出"API正常，问题在渲染层"，缩小排查范围
- 每次修复后都要完整验证，不能只看HTTP状态码

## 通知模板

客户端发现问题时的通知格式：
```
[问题] /concept 页面返回 Internal Server Error (500)
- API端点 /api/concepts: 正常(200)
- 页面端点 /concept: 错误(500)
- 问题定位: 前端渲染层
- 建议检查: 模板文件、路由配置
```

## 常见陷阱

### 端口占用导致网站不可达
443端口可能被OpenVPN或其他服务占用，Nginx被迫监听非标准端口（如8443）。访问时必须带端口号：`https://domain:8443/`。

**排查步骤**：
```bash
# 检查443被谁占用
ss -tlnp | grep 443
# 检查Nginx实际监听端口
grep -r "listen" /etc/nginx/sites-enabled/
```

**解决方案**：要么改OpenVPN端口让出443，要么用户接受带端口访问。
